## Description


Use when receiving code review feedback, before implementing suggestions. Requires technical rigor and verification, not performative agreement or blind implementation.

## When to Use

- Receiving code review feedback
- Evaluating review suggestions
- Deciding whether to implement feedback
- Clarifying unclear review comments
- Challenging technically questionable feedback

## When NOT to Use

- Accepting all feedback without question
- Implementing without understanding
- Dismissing all feedback
- Real-time code collaboration

## Workflow

### Phase 1: Understand Feedback

- Read all review comments
- Identify specific issues
- Note suggestions and concerns
- Ask for clarification if needed

### Phase 2: Evaluate Feedback

- Assess technical merit
- Consider alternatives
- Verify against requirements
- Check for conflicts

### Phase 3: Decide & Respond

- Accept valid feedback
- Discuss questionable suggestions
- Propose alternatives if needed
- Document decisions

### Phase 4: Implement & Verify

- Make approved changes
- Test thoroughly
- Verify fixes work
- Request re-review if needed

## Best Practices

- Ask for clarification on unclear feedback
- Verify feedback is technically sound
- Consider reviewer's perspective
- Test changes thoroughly
- Document decisions
- Respond respectfully
- Learn from feedback
