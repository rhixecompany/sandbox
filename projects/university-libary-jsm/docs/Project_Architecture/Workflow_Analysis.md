## Workflows

### Workflow 1: Book Borrowing Workflow

**Description:** Member searches for a book, borrows it, and the system tracks due dates with automated reminders.

**Entry Points:**
- **Pages:** Book catalog, detail, borrow forms
- **Actions:** `lib/actions/book.ts`, `lib/actions/user.ts`
- **Queries:** `lib/queries/select.ts`, `insert.ts`, `update.ts`, `delete.ts`

**Steps:**
1. User (member) browses/search book catalog
2. Server action queries database via Drizzle for matching books
3. User selects a book and views availability status
4. If available, user clicks "Borrow" → `borrowBook()` action
5. Action checks:
   - User membership is active and valid
   - Book is available (not already borrowed/reserved)
   - User has no overdue books exceeding limit
   - User hasn't reached max borrow limit
6. Action creates a borrowing record with due date (e.g., 14 days out)
7. Book status updated to "borrowed"
8. Redis updates cache for book availability and user's borrowed list
9. Confirmation shown to user
10. Background monitoring tracks due dates for reminders

**Error Handling:**
- Book already borrowed → clear message, suggest reservation
- User has overdue → block borrow, show overdue items
- Member not found → 404 or account creation prompt
- Validation via `lib/validations.ts` (Zod schemas)
- Redis connection failure → fallback to direct DB query

**Test Patterns:**
- Drizzle schema validation
- Server action unit tests
- E2E borrowing flow tests

### Workflow 2: Book Reservation & Due Date Management

**Description:** User reserves a currently-borrowed book and gets notified when available; system tracks overdue books with automated notifications.

**Entry Points:**
- `lib/actions/book.ts` — reservation actions
- `lib/workflow.ts` — business workflow logic
- Redis for session caching and rate limiting
- `lib/ratelimit.ts` — rate limiting for API actions

**Steps:**
1. User finds book is currently borrowed
2. User clicks "Reserve" to join waitlist
3. System records reservation with position in queue
4. When book is returned:
   - System checks next in reservation queue
   - Notification sent to reserving user
   - Reservation has a time window (e.g., 48 hours) to claim
5. If user doesn't claim within window, next in queue gets notified
6. Overdue tracking runs periodically:
   - System queries borrowed books past due date
   - Applies late fees or restrictions
   - Sends reminder notifications
7. Redis caches availability status for fast lookups

**Error Handling:**
- Reservation queue race condition → database transaction isolation
- Notification failure → retry mechanism
- Claim window expired → release to next in queue
