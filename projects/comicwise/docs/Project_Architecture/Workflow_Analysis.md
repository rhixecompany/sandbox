## Project Overview

**Project:** Comicwise

## Workflows

### Workflow 1: Browse, Search & Read Comic

**Description:** User discovers comics through browsing/search, views details, and reads chapters with progress tracking.

**Entry Points:**
- **Pages:** Home, Browse, Search, Genre pages
- **API:** Prisma queries for comic/chapter data
- **Actions:** `src/tests/e2e/pages/comics.spec.ts`, `comic-detail.spec.ts`

**Steps:**
1. User lands on home/browse page showing comic grid
2. Frontend fetches comics via Prisma (server component)
3. User filters by genre, author, or search query
4. Search/filter actions query database with filter criteria
5. User clicks on a comic → navigates to comic detail page
6. Detail page fetches comic metadata, chapters, ratings, comments
7. User selects a chapter → reader interface loads
8. Chapter images load with Next.js Image optimization
9. Reading progress is tracked (auto-save position)
10. User can bookmark, rate, or comment on the comic
11. User preferences and reading history are persisted

**Error Handling:**
- Comic not found → 404 page with similar comics
- Image load failure → fallback placeholder image
- Search with no results → "no comics found" message with suggestions
- Auth required for bookmarks → redirect to sign-in

**Test Patterns:**
- E2E tests: `browse.spec.ts`, `search.spec.ts`, `comic-detail.spec.ts`, `genre.spec.ts`

### Workflow 2: User Subscription & Stripe Payments

**Description:** User subscribes to premium tier via Stripe for ad-free reading and early chapter access.

**Entry Points:**
- **Pages:** Subscription/Pricing page
- **API:** Stripe webhooks integration
- **Database:** Prisma subscription model

**Steps:**
1. User browses subscription plans on pricing page
2. User selects plan and clicks "Subscribe"
3. Frontend calls API to create Stripe Checkout Session
4. User is redirected to Stripe Checkout for payment
5. Stripe processes payment (card, PayPal, etc.)
6. Stripe sends webhook to `/api/webhooks/stripe`
7. Backend verifies webhook signature
8. Backend updates user's subscription status in database
9. User is redirected back to app with success message
10. User now has premium access features enabled

**Error Handling:**
- Payment failure → Stripe handles retry, frontend shows error
- Webhook delivery failure → Stripe retries webhook
- Subscription expiry → background job via cron to downgrade
- Signature verification fail → reject webhook

**Test Patterns:**
- E2E: subscription flow testing with Stripe test mode
- Webhook handler unit tests
- Prisma schema validation for subscription status transitions
