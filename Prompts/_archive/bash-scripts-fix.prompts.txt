# Bash Scripts Fix

> Search, identify, catalog, and modernize all bash, PowerShell, bat, and TypeScript scripts across multiple repositories.

## Overview

This prompt orchestrates a comprehensive audit and modernization of all scripts in the workspace, consolidating functionality into TypeScript with AST-safe operations, dry-run support, and dead code elimination.

## Canonical Script Location Policy

- All operational scripts must live under `Bash/**`.
- Any script migrated from outside `Bash/**` must be deleted from its original location after parity validation.
- Exception: database seed scripts and related seed entry points may remain where framework conventions require them (for example npm script commands like `db:seed` or `seed`, and framework seed files).
- Do not keep duplicate script implementations in both `Bash/**` and non-`Bash/**` paths.

## Target Directories

- `Bash/scripts/**`
- `Bash/**`
- `projects/Banking/scripts/**`
- `projects/Banking/**`
- `projects/comicwise/scripts/**`
- `projects/comicwise/**`

- `brainstorming`

## Phase 1: Catalog All Scripts

**Goal:** Identify and document all scripts across target directories.

**Steps:**
1. Load required skills: `brainstorming`, `plans-and-specs`, `dispatching-parallel-agents`, `subagent-driven-development`, `systematic-debugging`, `simplify`
2. Search, identify, list, triage, and catalog all script types:
   - Bash scripts (`.sh`)
   - PowerShell scripts (`.ps1`)
   - Bat scripts (`.bat`)
   - Custom TypeScript scripts (`.ts`)
3. Create and update a summary report
4. Save the summary report to `docs/bash-scripts-list-context.md` formatted as markdown
5. Mark each script as one of:
   - Migrate to `Bash/**` and delete original
   - Keep in place (database seed exception only)
   - Delete as dead code

## Phase 2: Create Implementation Plan

**Goal:** Design a comprehensive modernization strategy with proposed fixes and enhancements.

**Steps:**
1. Read and understand `docs/bash-scripts-list-context.md`
2. Use `brainstorming` and `plans-and-specs` skills to create:
   - Implementation plan with features specs
   - Proposed fixes and enhancements with code samples and file references
3. Design architecture where:
   - All bash, PowerShell, and bat scripts are orchestrators only
   - All functionality is implemented in custom TypeScript scripts
   - TypeScript uses `ts-morph` for AST-safe operations
   - Dry-run functionality is built-in
4. Plan to:
   - Create/update and consolidate needed scripts into `Bash/**`
   - Delete all dead code
   - Delete original migrated scripts outside `Bash/**` (except database seed exception)
   - Update `package.json` accordingly
   - Validate all scripts are working

## Phase 3: Code Review and Fix

**Goal:** Review all bash scripts and fix identified issues.

**Steps:**
1. Code-review all bash scripts to identify:
   - Formatting inconsistencies (headings, lists, code blocks)
   - Content issues (outdated information, contradictions, unclear instructions)
   - Structural problems (organization, missing sections, redundancy)
2. Fix all identified issues
3. Verify scripts are properly formatted and functional
4. Enforce final placement and cleanup:
   - Confirm all non-exempt scripts live in `Bash/**`
   - Delete migrated originals outside `Bash/**`
   - Preserve only database seed exceptions (`db:seed`, `seed`, and required framework seed files)

## SAFETY & RISK MANAGEMENT

### CRITICAL: AST Transformation Validation Protocol
- **CRITICAL RISK**: ts-morph AST transformations could introduce subtle bugs in production scripts
- **Detailed Validation Required Before Any Transformation**:
  - [ ] Generate visual AST comparison: original vs transformed code trees
  - [ ] Run behavior-identical test suite on both versions
  - [ ] Verify semantic equivalence: identical inputs produce identical outputs
  - [ ] Maintain code coverage above 95% for all transformed code
  - [ ] **Require peer review for every AST transformation**
  - [ ] Document transformation rationale with before/after code samples
  - [ ] Create regression test suite for transformed functionality

### Dead Code Deletion Safety Protocol
- **CRITICAL**: Destructive deletion requires explicit multi-step confirmation
- **Migration Rule Override**: For scripts successfully migrated into `Bash/**`, delete the original non-`Bash/**` script in the same change set after parity checks pass.
- **Exception**: Do not force-delete framework-required database seed scripts (`db:seed`, `seed`, or required seed files/entry points).
- **30-Day Quarantine Process**:
  - Step 1: Tag dead code for deletion with comment: `// DEAD_CODE: marked for removal [DATE]`
  - Step 2: Wait 30 days for any late dependencies to surface
  - Step 3: Search codebase for any remaining references to marked code
  - Step 4: Require written justification for each file deletion
  - Step 5: Maintain deleted code archive for 6 months (in git history with tag)
  - Step 6: Notify all dependent scripts/tests before final deletion
  - Step 7: Require 2-person approval for irreversible deletion
  - Step 8: Create git commit with clear explanation of what was removed and why

### Dry-Run Fidelity Verification
- **Safety**: Dry-run execution must exactly mirror real execution
- **Verification Process**:
  - [ ] Compare dry-run output with real execution on identical test data
  - [ ] Verify dry-run captures all side-effect checks (file writes, API calls, etc.)
  - [ ] Document any differences between dry-run and real behavior
  - [ ] **Fail if dry-run behavior diverges from real behavior**
  - [ ] Test dry-run mode on new scripts before deploying real mode

### Behavior Parity & Change Tracking
- **Contract Preservation**: Keep script behavior parity unless explicitly approved
- **Approval Workflow**:
  - [ ] Create behavior test suite for each script (input→output mappings)
  - [ ] Run test suite on original and refactored versions
  - [ ] **Fail refactoring if behavior changes detected**
  - [ ] If changes necessary, document approved behavior changes with explicit rationale
  - [ ] Get 2-person sign-off for behavior modifications
  - [ ] Create migration guide for scripts with behavior changes

### TypeScript & Package.json Safety
- **Dependency Control**: Minimize package.json changes; justify every addition/removal
- **Injection Prevention**: Audit all TypeScript inputs for injection vulnerabilities
- **Build Verification**: Verify TypeScript compilation succeeds with no warnings
- **Performance Regression**: Monitor TypeScript script execution time vs original bash (fail if >20% slower)

## Alternatives

1. Alternative A: Incremental normalization only.
   - Keep most current wrappers and patch inconsistencies.
   - Pros: lower immediate change risk.
   - Cons: continued duplication and higher long-term maintenance cost.

2. Alternative B: Full central runner migration.
   - Route all wrapper behavior through shared TypeScript runners.
   - Pros: strongest consistency and testability.
   - Cons: larger short-term migration and coordination effort.

3. Recommended approach: Hybrid staged migration.
   - Start with high-impact wrapper normalization and dry-run standardization.
   - Migrate duplicated logic to TypeScript in controlled batches.
   - Best fit for existing mixed maturity across [Bash](Bash), [rhixecompany/Banking](rhixecompany/Banking), and [Rhixe-company/comicwise](Rhixe-company/comicwise).
