# enhance-markdown

## Description

`/enhance-markdown` is a four-phase markdown auditor and enhancer. It builds a two-way dependency catalog of the target file and all related files (Phase 1), creates a dual-channel fix plan (plugin system + companion markdown) and applies fixes in batches of 7 with Batch 1 as a proof-of-concept gate (Phase 2), executes any remaining plan items to completion (Phase 3), then independently verifies all fixes against the original issues list (Phase 4). Fully resumable — each phase checks for existing artifacts before re-running work.

---

## Usage

```bash
# Minimal — purpose derived from file frontmatter/first heading, fallback to filename
/enhance-markdown .opencode/commands/review.md

# Explicit purpose label — overrides inference
/enhance-markdown .opencode/commands/review.md review-command

# TXT → Markdown conversion mode (batch or single file)
/enhance-markdown --txt-to-md                    # batch all .txt in Prompts/
/enhance-markdown --txt-to-md Prompts/myprompt.txt  # single file
```

**Arguments:**

- `$1` — path to the target markdown file (required for audit mode)
- `$2` — purpose label slug (optional; overrides automatic inference)
- `--txt-to-md` — switch to TXT→Markdown conversion mode (optional)

**Error — file not found:** If `$1` does not resolve to an existing file, halt immediately and output: `Error: file not found — {$1}`

**Purpose resolution** (resolved once at start, used for all artifact names):

| Priority | Source | Example result |
| --- | --- | --- |
| 1 | Explicit `$2` argument | `review-command` |
| 2 | Semantic slug from frontmatter/first heading | `code-review-standards` |
| 3 | Filename without extension | `review` |

Slug-ification: lowercase → spaces and `_` → `-` → strip non-alphanumeric (except `-`) → truncate at 40 chars.

**Artifact map:**

| Artifact | Written by | Description |
| --- | --- | --- |
| `docs/{purpose}-context.md` | Phase 1 | Two-way dependency catalog |
| `docs/{purpose}-issues-context.md` | Phase 1 | Audit findings |
| `thoughts/plans/{purpose}-debug.md` | Phase 2 | Companion plan markdown |
| `docs/{purpose}-fix-issues-context.md` | Phase 2 | Fix plan + progress log |
| `docs/{purpose}-verify-context.md` | Phase 4 | Verification report |

Plugin planning system also registers plan and specs under namespace `{purpose}-debug`.

## TXT → Markdown Conversion Mode

Triggered when `$1` is `--txt-to-md`. Converts raw `.txt` prompt files in `Prompts/` to structured `.md` via a two-step sequential pipeline: enhance (Stanford/Anthropic patterns) → apply 9-section template.

### Mode Detection

Check `$ARGUMENTS`:

- **Argument present after `--txt-to-md`** → single-file mode. Target list = the specified `.txt` file path.
- **No argument after `--txt-to-md`** → batch mode. Target list = all `.txt` files found in `Prompts/` (non-recursive, `*.txt` glob).

If batch mode and no `.txt` files are found in `Prompts/`, print:

```
No .txt files found in Prompts/. Nothing to do.
```

…and exit cleanly.

### Per-File Pipeline (Phases 1–5)

Run the following phases for each file in the target list:

#### Phase 1 — Read

Read the full raw content of the target `.txt` file into context.

**Error**: If the file is unreadable, log:

```
ERROR: Could not read <filename> — skipping.
```

Increment the failed count and continue to the next file.

#### Phase 2 — Enhance

1. Read `.opencode/commands/prompt-engineering-prompt-enhancer.md` into context.
2. Apply its enhancement instructions **inline** to the raw content from Phase 1 — the enhancer applies Stanford/Anthropic patterns:
   - Critical rules positioned in the first 15% of the prompt
   - Nesting depth ≤ 4 levels
   - Instruction ratio 40–50%
   - Single source of truth (no rule repeated > 2×)
   - Explicit 3-tier prioritization (Safety → Core Workflow → Optimization)
3. Hold the **enhanced text** in context. Do NOT write any files at this stage.

**Error**: If the enhancer produces empty or clearly invalid output, log:

```
WARNING: Enhancer returned empty output for <filename> — skipping.
```

Increment the failed count and continue.

#### Phase 3 — Template

Apply the 9-section template below to the enhanced text from Phase 2. Populate each section by extracting and restructuring content from the enhanced text — **do not invent content**. Every populated section must trace back to something in the enhanced text.

##### Section order (fixed)

`Skills → Subagents → Personas → Rules → Phases → Steps → Tasks → Subtasks → Actions`

Only include a section if it passes its detection rule (see table below). Absent optional sections are omitted entirely. The 4 core sections are always present.

##### Core 4 — always present

```
## Rules
- <constraint or behavioral guardrail extracted from the enhanced text>

## Steps
1. <high-level ordered step>

## Tasks
- [ ] <discrete actionable unit of work>

## Actions
- `<tool or command>` — <what it does in this context>
```

If the enhanced text genuinely has nothing applicable for a core section, write `- N/A` rather than leaving it empty.

##### Optional 5 — include when detection rules match

```
## Skills
- `<skill-name>` — <why it applies to this prompt>

## Subagents
- `@<agent>` — <delegation scope>

## Personas
> <role definition / identity the AI adopts when running this prompt>

## Phases
### Phase 1: <name>
<what happens in this phase>

## Subtasks
- [ ] <sub-unit of a parent Task>
```

##### Detection rules for optional sections

| Section | Include when... |
| --- | --- |
| `Skills` | Prompt references tools, capabilities, or skill invocations |
| `Subagents` | Prompt involves delegation, parallel work, or specialist agents |
| `Personas` | Prompt defines a role, character, or identity for the AI |
| `Phases` | Workflow has 2+ distinct sequential stages |
| `Subtasks` | Any Task entry requires further decomposition |

#### Phase 4 — Write

Write the templated output from Phase 3 to `Prompts/<same-stem>.md`. Always overwrite if the file already exists.

- **Same-stem rule**: `Prompts/myprompt.txt` → `Prompts/myprompt.md`

**Error**: If the write fails, log:

```
ERROR: Could not write <output-filename>.
```

Increment the failed count and continue.

#### Phase 5 — Report (runs once, after all files complete)

Print:

```
Refactor complete.
  Processed: <n>
  Failed:    <n>  (see errors above)
  Total:     <n>
```

### @analyst _(Phase 1 — read-only investigator)_

Delegates to: `codebase-locator` (forward/reverse scan) and `codebase-analyzer` (batch analysis). Writes output artifacts in `docs/`.

**Constraints:**

- Read-only: no edits except writing output artifacts to `docs/`
- Complete the full two-way scan **before** any batch-reading begins
- Batch-read in groups of exactly 7 (fewer only if the remaining set is smaller)
- Run up to 3 concurrent batches of 7 in parallel
- Report every issue found — no filtering or severity thresholding at this stage

**Skills:** `plans-and-specs`, `systematic-debugging`

Delegates to: `opencoder` for plan authoring and fix application; `openagent` for agent enablement checks.

**Constraints:**

- Run agent readiness check before any fixes
- Apply fixes in batches of exactly 7 files, sorted Critical → Major → Minor
- Batch 1 is a proof-of-concept — verify quality before continuing to Batch 2
- Append each batch result to `docs/{purpose}-fix-issues-context.md` under `## Progress > Batch N`
- On batch failure: log failure and continue — do not halt

**Skills:** `plans-and-specs`, `writing-plans`, `simplify`

### @orchestrator _(Phases 3 & 4 — completion engine and independent verifier)_

Delegates to: `openagent` for Phase 3 execution.

**Constraints:**

- Phase 3: read plugin plan first (`readPlan`), fall back to `thoughts/plans/{purpose}-debug.md` if plugin system unreachable
- Phase 3: skip any plan item already marked `[x]` — never re-apply completed work
- Phase 3: call `markPlanDone("{purpose}-debug")` only after zero remaining items
- Phase 4: read `docs/{purpose}-context.md` and `docs/{purpose}-issues-context.md` **only** before verifying — do not read Phase 2/3 outputs first (prevents confirmation bias)
- Phase 4: write `docs/{purpose}-verify-context.md` with pass/fail per original issue

**Skills:** `plans-and-specs`, `verification-before-completion`

## Rules

1. Source file (`$1`) is **never deleted** — only edited in-place when it is a fix target
2. Phase entry checks **must** run before any phase work — skip phases whose output artifacts already exist
3. Batch size is exactly **7 files** — never more, fewer only when the remaining set is smaller
4. Batch 1 is a **proof-of-concept gate** — Phase 2 halts on Batch 1 failure (Batch 2+ continue on failure)
5. All artifact writes go to `docs/` or `thoughts/plans/` — no writes to `.opencode/` except via `opencoder`
6. Phase 4 reads **only Phase 1 outputs** before verifying — Phase 2/3 outputs read only after initial check (prevents confirmation bias)
7. Plugin planning system is **optional** — companion markdown is always written regardless
8. `markPlanDone` is called **only after zero remaining `[ ]` items** in the companion plan

## Phase 1 — Catalog & Audit

**@analyst** runs this phase.

### Entry Check

```
IF docs/{purpose}-issues-context.md EXISTS → Phase 1 complete, skip to Phase 2
ELSE IF docs/{purpose}-context.md EXISTS  → catalog exists, skip to Step 1.3
ELSE                                       → run full Phase 1
```

### Step 1.1 — Purpose Resolution

Resolve `{purpose}` using cascading priority:

1. `$2` argument if provided
2. Slug from file's frontmatter `description` or first `# heading` — lowercase, spaces/`_` → `-`, strip non-alphanumeric except `-`, truncate at 40 chars
3. Filename without extension

Use the resolved `{purpose}` for all artifact names in this and subsequent phases.

### Step 1.2 — Two-Way Dependency Scan

**Forward scan** (what the file references):

- Extract all markdown links `[text](path)` pointing to `.md` files
- Extract all `@agent-name` mentions
- Extract all `/command-name` trigger references
- Extract all plan/spec namespace slugs (patterns like `{slug}-debug`, `{slug}-context`)
- Query plugin system (`readPlan`) for matching namespaces

**Reverse scan** (what references this file):

- `grep` all `.md` files in `.opencode/` and `docs/` for this file's path, filename, and trigger
- Query plugin system for plans/specs that reference this file by name

**Write `docs/{purpose}-context.md`:**

```markdown
# {purpose} — Context Catalog

Generated: {date} Source file: {path}

## Summary

{2-3 sentence description of the file's role and relationships}

## Forward References (this file → others)

| Type | Reference | Path | Exists? |
| ---- | --------- | ---- | ------- |

## Reverse References (others → this file)

| Type | Source File | Reference | Context snippet |
| ---- | ----------- | --------- | --------------- |

## Plans & Specs (plugin system)

| Namespace | Type | Status | Linked to source? |
| --------- | ---- | ------ | ----------------- |

## File Inventory (all related files to audit)

| #   | Path | Type | Priority |
| --- | ---- | ---- | -------- |
```

**Priority assignment for File Inventory:**

- `Critical` — the source file itself (`$1`)
- `Major` — files directly referenced by the source file (forward references)
- `Minor` — files that reference the source file only (reverse references)

### Step 1.3 — Parallel Batch-Reading

Extract all paths from `## File Inventory`. Group into batches of 7. Run up to 3 concurrent batches.

For each file, checking:

- **Formatting:** heading hierarchy violations, inconsistent list styles, unclosed code fences, broken tables, mixed indentation
- **Content:** outdated version references, contradictions with related files, unclear/ambiguous instructions, missing required sections (frontmatter `trigger`/`description`, standard section headings for file type)
- **Structural:** missing standard sections for this file type, redundant content, illogical section order
- **Cross-file:** references to non-existent files, mismatched terminology across related files

### Step 1.4 — Issues Report

**Write `docs/{purpose}-issues-context.md`:**

```markdown
# {purpose} — Issues Catalog

Generated: {date} Files audited: {N} across {B} batches

## Summary

Total: {N} | Critical: {N} | Major: {N} | Minor: {N}

## Issues by File

### {file-path}

#### Formatting Issues

- [ ] {issue-id} — Line {N}: {description}

#### Content Issues

- [ ] {issue-id} — {description}

#### Structural Issues

- [ ] {issue-id} — {description}

## Cross-File Issues

- [ ] {issue-id} — affects {file-a}, {file-b}: {description}

## Statistics

| Category   | Count |
| ---------- | ----- |
| Formatting | {N}   |
| Content    | {N}   |
| Structural | {N}   |
| Cross-file | {N}   |
```

Before authoring fixes, verify the plugin planning system is reachable:

```
TRY   createPlan("{purpose}-debug", "enhance-markdown dual-channel plan")
  → plugin system available: proceed with plugin + companion markdown
CATCH → plugin system unreachable: proceed with companion markdown only
```

**Create companion markdown** (`thoughts/plans/{purpose}-debug.md`):

```markdown
# {purpose} — Fix Plan

Generated: {date} Source: docs/{purpose}-issues-context.md

## Objective

{1-2 sentence description of what needs to be fixed}

## Fix Items

Ordered Critical → Major → Minor, matching issue IDs from Phase 1.

### Batch 1 — Proof of Concept (≤7 items)

- [ ] {issue-id}: {file} — {description} [Priority: Critical/Major/Minor]

### Batch 2 (≤7 items)

<!-- append batch results here -->
```

**If plugin system available, also:**

- `createSpec("{purpose}-batch-1", "Batch 1 proof-of-concept fixes")` — include first ≤7 issue items

**Write `docs/{purpose}-fix-issues-context.md`** with header only (batches appended in 2.3):

```markdown

Generated: {date} Total issues: {N} | Batches planned: {B}

## Plan Reference

Plugin plan: {purpose}-debug (or "unavailable — using companion markdown only") Companion plan: thoughts/plans/{purpose}-debug.md
```

### Step 2.3 — Fix Batches

Process batches in order Critical → Major → Minor, ≤7 files per batch.

**Batch 1 is a proof-of-concept:**

1. Apply fixes for the first ≤7 items via `opencoder`
2. Re-read all modified files and verify each fix was applied correctly
3. IF Batch 1 status line reads "✅ Passed" → continue to Batch 2
4. IF Batch 1 status line reads "❌ Failed" → log failure reason, halt Phase 2, surface for user review

**Batch 2+ (standard):**

1. Apply fixes for the next ≤7 items via `opencoder`
2. Spot-check at least 2 files per batch (first and last)
3. Mark each item `[x]` in companion markdown as completed
4. Append batch result to `docs/{purpose}-fix-issues-context.md`

**On any batch failure (Batch 2+):** log failure, mark affected items `[ ]` still open, continue to next batch — do not halt.

### Step 2.4 — Fix Progress Log

After every batch, append to `docs/{purpose}-fix-issues-context.md`:

```markdown
## Progress > Batch {N}

Status: ✅ Passed | ❌ Failed Items attempted: {list of issue-ids} Items completed: {list of issue-ids} Items failed: {list with reason} Notes: {any relevant observations}
```

After all batches, append final summary:

```markdown
## ✅ All batches complete

Total fixed: {N}/{total} Remaining open: {N} (list issue-ids if any)
```

## Phase 3 — Execute

**@orchestrator** runs this phase.

### Entry Check

```
IF docs/{purpose}-fix-issues-context.md CONTAINS "## ✅ All batches complete"
  AND thoughts/plans/{purpose}-debug.md has zero unchecked items [ ]
    → Phase 3 complete, skip to Phase 4
ELSE
    → run Phase 3
```

### Step 3.1 — Load & Reconcile State

Determine current state before executing anything:

1. **Load plan** — try plugin system first:

   ```
   TRY   readPlan("{purpose}-debug")
   CATCH → read thoughts/plans/{purpose}-debug.md directly
   ```

2. **Reconcile** — cross-reference plan items against `docs/{purpose}-fix-issues-context.md` progress log:
   - Items marked `[x]` in plan AND in progress log → **confirmed complete**, skip
   - Items marked `[x]` in plan but absent from progress log → **unconfirmed**, re-verify before skipping
   - Items marked `[ ]` in plan → **pending**, queue for execution

3. **Report state** (log to session, do not write to file):
   ```
   State: {N} complete | {N} pending | {N} unconfirmed
   ```

### Step 3.2 — Execute Remaining Items

For each pending item (in order Critical → Major → Minor):

1. Apply fix via `openagent`
2. Immediately re-read the modified file and verify the fix is present
3. Update plan item from `[ ]` to `[x]` in companion markdown
4. Append to `docs/{purpose}-fix-issues-context.md` under existing `## Progress > Batch {N}` section (or create `## Progress > Phase 3 Catchup` if no batch section applies)

**On failure:** log to progress log with reason, leave item `[ ]`, continue to next item.

**Unconfirmed items:** re-read target file. If fix is already present → mark `[x]` with note `(confirmed present)`. If absent → re-apply and verify.

### Step 3.3 — Plan Completion

After zero pending items remain:

1. Verify companion markdown has no unchecked `[ ]` items
2. If plugin system was available: call `markPlanDone("{purpose}-debug")`
3. Append to `docs/{purpose}-fix-issues-context.md`:
   ```markdown
   ## ✅ Phase 3 Complete

   All items resolved. Plan marked done. Remaining open (if any): {list issue-ids or "none"}
   ```
